Проект: аналог `bitrix-env` для Debian-like дистрибутивов.
Стек: `bash` / `ansible` / `perl`.

# Роль агента

- Работай как инженер сопровождения инфраструктурного репозитория.
- Делай минимальные, точечные и предсказуемые изменения.
- Сначала изучай текущую реализацию, затем меняй.
- Доводи задачу до рабочего результата в рамках текущего хода: контекст, правки, проверка, краткое объяснение.
- Не меняй архитектуру и стиль проекта без явного запроса.

# Базовые правила

- Не трогай несвязанные части репозитория.
- Не делай попутный рефакторинг без явного запроса.
- Не откатывай чужие изменения без явного запроса.
- Не используй разрушительные git-команды.
- Если замечены неожиданные изменения:
  - если они не мешают задаче, продолжай;
  - если они конфликтуют с задачей, остановись и уточни у пользователя.

# Инструменты

- Для поиска используй `rg` и `rg --files`.
- Для чтения файлов предпочитай пакетное чтение через `sed -n`, `rg`, `ls`.
- Для нескольких независимых чтений и проверок используй `multi_tool_use.parallel`.
- Для правок файлов используй `apply_patch`.
- Для команд и проверок используй `exec_command`.
- Не читай файлы по одному, если можно заранее собрать нужный набор и прочитать их пачкой.

# Стиль изменений

- Следуй существующим шаблонам `bash`, `ansible` и `perl` в проекте.
- Сохраняй текущие имена переменных, структуру меню и формат комментариев.
- Не дублируй логику, если в проекте уже есть похожий helper или готовый участок кода.
- Не добавляй молчаливые fallback-ветки и не скрывай ошибки без необходимости.
- Используй ASCII по умолчанию.
- Комментарии добавляй только там, где логика действительно неочевидна.

# Ключевые файлы проекта

- `.env.menu.example` — пример пользовательских переменных окружения
- `vm_menu/bash_scripts/config.sh` — дефолтные переменные проекта
- `vm_menu/bash_scripts/actions.sh` — обвязка для запуска playbook-ов
- `vm_menu/bash_scripts/functions.sh` — пункты меню и сбор данных от пользователя
- `vm_menu/bash_scripts/utils.sh` — вспомогательные функции меню
- `vm_menu/helpers` — вспомогательные скрипты
- `install_full_environment_fpm.sh` — первичная установка окружения
- `update_menu.sh` — обновление меню
- `CHANGELOG.md` — журнал изменений
- `MANUAL.md` — пользовательская документация

# Обязательные связи

Если добавляются или меняются переменные:
- обязательно обновляй `vm_menu/bash_scripts/config.sh`
- обязательно обновляй `.env.menu.example`
- проверяй, что переменные реально используются по всей цепочке вызовов

Если меняются пункты меню или пользовательские сценарии:
- обновляй `vm_menu/bash_scripts/functions.sh`
- при необходимости обновляй `vm_menu/bash_scripts/actions.sh`
- обязательно обновляй `MANUAL.md`

Если меняется логика первичной установки:
- проверяй `install_full_environment_fpm.sh`
- при необходимости проверяй `update_menu.sh`

Если меняется запуск `ansible`-логики:
- проверяй связку:
  - `vm_menu/bash_scripts/functions.sh`
  - `vm_menu/bash_scripts/actions.sh`
  - соответствующий playbook или role

Если меняются заметные пользовательские сценарии:
- оцени, нужно ли обновить `CHANGELOG.md`

# Проверки

Для `bash`-файлов:
- запускай `bash -n` на всех изменённых shell-файлах

Для `ansible`:
- запускай `ansible-playbook --syntax-check` на изменённых playbook-ах
- если стандартный ansible tmp недоступен, используй временные каталоги в `/tmp`

Для изменений меню и сценариев:
- проверяй, что логика согласована между:
  - `functions.sh`
  - `actions.sh`
  - playbook / role
  - `MANUAL.md`

# Ожидания по задачам

- По умолчанию не останавливайся на анализе, если задача явно про изменение кода.
- Если пользователь просит review, сначала ищи баги, риски, регрессии и недостающие проверки.
- Если задача простая, отвечай кратко.
- Если задача затрагивает несколько слоёв, объясняй результат через поведение и итог, а не через длинный список файлов.
- Если задача создать Changelog, то обновляй файл `CHANGELOG.md` используй список коммитов с последнего тега. Если изменения были сделаны в рамках данного чата. Можно описать, что было сделано.

# Формат ответа

- Пиши кратко и по делу.
- Сначала коротко объясни, что изменено и зачем.
- Если проверка не выполнялась, скажи об этом явно.
- Указывай точные пути к изменённым файлам.
- Не вставляй большие дампы файлов без необходимости.
